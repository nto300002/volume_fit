# 開発計画

## 実装順序

### Phase 1: Web 縦切り

```text
Googleまたはメールでログイン
↓
初回プロフィール
↓
腕立て伏せ1セット入力
↓
Firestoreへ保存
↓
推定負荷・ボリューム表示
↓
AI Markdown生成
↓
AI出力履歴保存
```

実装項目:

- Flutter Web 基盤
- Firebase 接続
- 認証
- プロフィール
- 腕立て伏せ 1 セット記録
- 保存
- 計算
- AI 出力

### Phase 2: Web MVP

- 複数セット
- 前セット複製
- 外部重量
- 履歴
- 比較
- 計算設定
- 次回予定
- JSON 出力
- アカウント削除

### Phase 3: PWA

- インストール可能
- オフライン起動
- Firestore 保留書き込み表示
- 更新通知
- 安全な再読み込み

### Phase 4: Android

- 共通 Domain・Application・Repository 契約を利用
- モバイル専用画面
- 片手操作
- OS 共有シート
- バックグラウンド移行時保存
- 下書き復元
- 内部テスト

## TDD 方針

各 Issue で以下を行う。

```text
Red
↓
Green
↓
Refactor
↓
自動テスト
↓
手動テスト
↓
結果記録
```

### Unit テスト

- 推定負荷
- ボリューム
- RIR 補正
- ハードセット
- 対象筋配分
- バリデーション
- Markdown 生成
- JSON 生成

### Repository テスト

Firebase Emulator を利用する。

- 保存
- 取得
- 更新
- 論理削除
- 復元
- ユーザー分離
- revision 競合

### Widget テスト

- ログイン
- 登録
- 入力フォーム
- RIR
- 計算表示
- 保存状態
- エラー
- AI プレビュー

### Integration テスト

最初の縦切り:

```text
ログイン
→ プロフィール設定
→ 腕立て伏せ1セット
→ Firestore保存
→ 計算結果表示
→ AI Markdown生成
→ AI履歴保存
```

## Issue 運用要件

### Issue 単位

1 Issue は小さなユーザー価値または確認可能な振る舞いとする。

悪い例:

```text
認証機能を全部実装する
```

良い例:

```text
メールアドレスとパスワードでログインできる
```

### Issue テンプレート

```md
## 概要

このIssueで実現するユーザー価値を記載する。

## 対象範囲

- 対象画面
- Domain
- Repository
- Firebase
- 対象外

## 仕様

### 正常系

### 異常系

### 境界値

## TDD

- [ ] 失敗するテストを先に追加した
- [ ] 最小実装でテストを成功させた
- [ ] リファクタリング後も成功する
- [ ] Unit／Widget／Repositoryテストを追加した
- [ ] 必要なIntegrationテストを追加した

## 受け入れ要件

- [ ] 関連する自動テストがすべて成功する
- [ ] `flutter analyze` が成功する
- [ ] Web実ブラウザで手動テストした
- [ ] 正常系を手動確認した
- [ ] 異常系を手動確認した
- [ ] 境界値を手動確認した
- [ ] レスポンシブ表示を確認した
- [ ] コンソールに未対応エラーがない
- [ ] 手動テスト結果をIssueまたはPRへ記録した
- [ ] 未確認事項が残っていない

## 手動テスト

| No. | 操作 | 期待結果 | 結果 |
|---:|---|---|---|
| 1 |  |  | 未実施 |

## 完了条件

実装、自動テスト、手動テスト、結果記録がすべて完了していること。
```

### Definition of Done

```text
実装完了
+
自動テスト成功
+
手動テスト成功
+
結果記録
```

手動テストが未実施なら Issue を閉じない。

## MVP 完成条件

### Web/PWA

```text
ログイン
↓
プロフィール設定
↓
外部重量・自重を記録
↓
Cloud Firestoreへ保存
↓
履歴確認
↓
比較
↓
AI出力
↓
AI履歴保存
↓
次回予定
↓
JSON出力
↓
アカウント削除
```

### Android

```text
同一アカウントでログイン
↓
Webの記録を確認
↓
トレーニングを入力
↓
Firestoreへ保存
↓
AI出力をOS共有
```

全 Issue について、自動テスト・手動テスト・結果記録が完了していることを MVP 完成条件とする。

## 根拠

- 保存先を Cloud Firestore へ一本化する確定事項を反映する。
- UI と Domain から Firebase SDK を分離するため、Repository 抽象化を維持する。
- Web 先行で入力体験と AI 出力価値を検証し、その後 Android へ展開する。
- TDD だけでなく Issue 単位の実ブラウザ手動テスト完了までを受け入れ要件に含める。
